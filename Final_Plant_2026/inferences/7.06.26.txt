From the beginning, we started with almost no TD3 implementation and only the Caterpillar Simulink plant model consisting of a reference input, a PI controller, a transfer function plant, and disturbance inputs. First, we studied the competition requirement and understood that the final controller should not use fixed PI gains but should adapt them using correction factors α and β, where (K_p = \alpha K_{p,base}) and (K_i = \beta K_{i,base}), while obeying the competition rule that the adaptive controller should depend only on the control error and Disturbance A. We then decided to keep the observation space simple and compliant by choosing only two inputs to the RL agent: ([error, A]), and two outputs: ([alpha, beta]). After that, we built the TD3 architecture from scratch in MATLAB by creating `obsInfo` and `actInfo`, then designing an Actor network whose job is to map the current state ([error,A]) into gain correction factors ([alpha,beta]). We then constructed two Critic networks, learning why TD3 uses twin critics to prevent the overestimation problem present in DDPG. After successfully creating the Actor and Critic objects, we created the actual TD3 agent object (`TheTD3Brain`) and connected it to the RL Agent block inside Simulink. At this stage we faced the first major problem: the RL Agent block continuously complained that `agentObj does not exist`. We solved this by understanding that the Simulink RL Agent block does not create a brain itself but instead expects a MATLAB workspace variable containing the trained RL agent. Once `TheTD3Brain` was created and linked to the RL Agent block, the error disappeared. Next, we created the RL environment using `rlSimulinkEnv`, connected the model `ProblemState` with the RL Agent block, and verified that the observation and action dimensions matched our design (2 inputs and 2 outputs). After this, we began training and immediately encountered numerical instability where the transfer function and later the PI integrator state became non-finite, producing huge values such as (3.22 \times 10^{28}). By investigating the signal paths and gain structure, we discovered that the controller gains were exploding, causing the integrator to blow up. We then analyzed the PI structure, verified how α and β were connected, and understood that TD3 outputs must remain bounded and should be treated as gain correction factors rather than directly becoming uncontrolled PI gains. Once the system became stable enough to train, we ran our first TD3 training session using a simple reward function (reward = -|error|). Training completed successfully, proving that the complete RL loop—observations, actions, reward calculation, actor, critics, replay buffer, TD3 learning process, and Simulink integration—was functioning correctly. We then learned what episodes, steps per episode, replay buffers, actor updates, critic updates, and sample times actually mean. We discovered that with a TD3 sample time of 0.1 seconds and a 1000-second simulation, a single episode contains 10,000 decision steps, which is computationally expensive. This led us to investigate whether the disturbance profile or TD3 sample time should be modified. We examined the disturbance generation subsystem and found that Disturbance A is computed using (A = 20 \times disturbance + 15), meaning the disturbance profile already exposes the agent to varying operating conditions. We also clarified the difference between training disturbances and the disturbance percentages (5%, 10%, 15%, 20%, and 30%) mentioned in the Caterpillar brochure, concluding that the brochure percentages are likely evaluation conditions while the current disturbance profile can still be used effectively during training. We then discussed reward engineering in depth, comparing the simple (-|error|) reward with a much richer reward function that considers improvement, overshoot, settling time, steady-state accuracy, gain thrashing, and disturbance difficulty. We learned that reward design is often more important than the choice of RL algorithm itself because the reward function defines the behavior that TD3 will ultimately learn. Finally, we verified workspace persistence by saving and reloading the complete TD3 agent using `.mat` files, ensuring that all progress can be restored across MATLAB sessions. At the current stage, we have a fully functioning TD3 training framework with a working Simulink environment, Actor network, twin Critic networks, replay buffer, training pipeline, reward structure, disturbance model, and saved agent. The remaining work is no longer infrastructure building but rather performance optimization: refining the reward function, selecting an appropriate TD3 sample time, designing robust disturbance training strategies, logging α and β behavior, and demonstrating that the adaptive TD3-tuned PI controller outperforms a fixed PI controller under the Caterpillar disturbance scenarios.
